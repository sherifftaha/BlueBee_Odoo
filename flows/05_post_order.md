# 📦 Sub-Flow #5: ما بعد الأوردر (Post-Order)

> **Project:** BlueBee-Eg B2B Wholesale Platform
> **Module:** `invoice_deadline` (Odoo 17)
> **Phase:** 1 — UX Planning
> **Status:** 🟡 Draft — في انتظار مراجعة شريف
> **Date:** June 2026
> **Scope:** يغطي كل ما يخص الفاتورة بعد ما اتولدت — من لوحة الفواتير (بكل حالاتها)، لصفحة الفاتورة المفردة (تفاصيل + أكشن + سبب الرفض + العداد)، لحالة الشحن البسيطة بعد الدفع، لقسم المرتجع (الديفوهات). بيشمل عرض حالات Open / Locked / Grace / Continuation / Blocked / Paid من جهة التاجر.
> **Scope note:** Covers everything about the invoice after it's created — from the invoices dashboard (all states), to the single invoice page (details + actions + rejection reason + counter), to the simple post-payment shipment status, to the returns (defects) section. Includes the display of Open / Locked / Grace / Continuation / Blocked / Paid states from the merchant's side.

<a name="الهدف"></a>
## 🎯 الهدف من الـ Flow | Flow Goals

ما بعد الأوردر هو **مساحة المتابعة والطمأنينة** — اللحظة اللي التاجر بيرجع فيها يتابع التزاماته ويعرف هو واقف فين. الهدف:

Post-order is **the follow-up and reassurance space** — where the merchant returns to track commitments and know where things stand. Goals:

1. **يعرف كل فواتيره وحالاتها في مكان واحد** — لوحة واضحة، كل فاتورة وحالتها والعداد بتاعها
   See all invoices and states in one place — clear dashboard, each invoice with its state and counter

2. **يفهم هو واقف فين في كل فاتورة** — Open بيضيف؟ Locked محتاج يدفع؟ Grace خلاص قرب البلوك؟
   Understand where he stands in each invoice — Open adding? Locked needs payment? Grace nearing block?

3. **يلاقي طريقه للأكشن الصح** — اطلب دفع، أضف قطع، تابع الشحن، أبلغ عن ديفو
   Find the right action — request payment, add pieces, track shipment, report a defect

4. **يطمّن إن طلبه ماشي بعد الدفع** — حالة شحن بسيطة وواضحة (بيتجهز / اتشحن)
   Reassurance the order is moving after payment — simple, clear shipment status (preparing / shipped)

5. **يعرف يعمل إيه لو فيه ديفو** — سياسة المرتجع واضحة + طريق مباشر للتواصل
   Know what to do if there's a defect — clear returns policy + direct contact path

<a name="النطاق"></a>
## 🔗 النطاق والربط مع باقي الـ Flows | Scope & Connections

هذا الـ Sub-Flow بيبدأ من **وجود فاتورة (أي حالة)** وبيغطي كل تفاعل التاجر معاها بعد إنشائها، لحد استلام البضاعة وما بعده (المرتجع).

This Sub-Flow starts from **an existing invoice (any state)** and covers all merchant interaction with it after creation, through delivery and beyond (returns).

| المرحلة Stage | الوصف Description |
|---|---|
| لوحة الفواتير | قائمة بكل فواتير التاجر + حالة كل واحدة + العداد |
| صفحة الفاتورة | تفاصيل القطع، الحالة، العداد، الأكشن، سبب الرفض لو موجود |
| حالة الشحن | بعد القبول للشحن: بيتجهز / اتشحن (بدون رقم تتبع) |
| المرتجع | عرض سياسة الديفوهات + زر تواصل واتس (بدون فورم) |

### الربط مع Sub-Flows التانية | Connections to Other Sub-Flows

- **Sub-Flow #3 (Cart):** الـ Confirm بيولّد فاتورة Open → بتظهر هنا في اللوحة
- **Sub-Flow #4 (Payment):** "اطلب دفع" من صفحة الفاتورة بيدخل على flow الدفع. ونتيجة المراجعة (قبول/رفض) بترجع تتعرض هنا
- **Sub-Flow #1 (Home):** "فاتورتي" في الـ Navbar بيوصّل للوحة الفواتير

> **حد فاصل مهم:** الـ Sub-Flow ده بيعرض الفواتير وحالاتها والأكشن المتاح. **عملية الدفع نفسها** (اختيار المسار + رفع الإيصال + المراجعة) موضوع Sub-Flow #4. هنا بنعرض زر "اطلب دفع" وبنستقبل النتيجة بس.
>
> **Important boundary:** This Sub-Flow displays invoices, states, and available actions. **The payment process itself** (path choice + receipt upload + review) belongs to Sub-Flow #4. Here we show the "Request Payment" button and receive the outcome only.

<a name="القرارات-المعمارية"></a>
## ✅ القرارات المعمارية | Architectural Decisions

| # | القرار Decision | الاختيار Choice | السبب Rationale |
|---|---|---|---|
| 1 | لوحة الفواتير Invoices dashboard | **قائمة موحّدة لكل الفواتير، مفلترة بالحالة** (الكل / مفتوحة / محتاجة دفع / مدفوعة) | التاجر يلاقي كل حاجة في مكان واحد، الفلتر يساعده يركّز |
| 2 | ترتيب الفواتير Sort order | **الأحدث والأكثر إلحاحاً فوق** — الفواتير اللي محتاجة أكشن (Locked/Grace) في القمة | الإلحاح أهم من التاريخ — التاجر لازم يشوف اللي بيقرب يبلوك أولاً |
| 3 | حالة الشحن Shipment status | **بسيطة: بيتجهز / اتشحن فقط** — بدون رقم تتبع ولا شركة شحن | الشركة بتشحن يدوي (مندوب + بريد)، مفيش tracking number حقيقي |
| 4 | المرتجع Returns | **عرض السياسة + زر تواصل واتس** — بدون فورم رفع صور | السياسة بتتطلب فحص بشري للصور خلال 24 ساعة، الفورم يرجع لخدمة العملاء في الآخر |
| 5 | العداد Counter | **عداد ظاهر في صفحة الفاتورة + اللوحة** للفواتير المفتوحة (Open/Continuation) | التاجر لازم يعرف باقي كام يوم قبل ما تتقفل |
| 6 | عرض الحالات States display | **Badge ملوّن لكل حالة + شرح بلغة التاجر** (مش لغة مطوّر) | "محتاجة دفع" مش "Locked"، "بتقرب تتقفل" مش "Grace" |
| 7 | صفحة الفاتورة المفردة Single invoice page | **صفحة كاملة** (مش modal) — تفاصيل + أكشن + تاريخ | الفاتورة كيان مهم، تستاهل صفحة بـ URL ثابت يتشارك مع الموظفين |
| 8 | الأكشن المتاح Available actions | **بيتغيّر حسب الحالة** — Open: أضف/اطلب دفع، Locked: اطلب دفع، Paid: تابع الشحن | كل حالة ليها أكشن منطقي، إخفاء غير المتاح يقلل الحيرة |
| 9 | سبب الرفض Rejection reason | **دائم في صفحة الفاتورة** لحد ما يدفع تاني (من Sub-Flow #4) | التاجر يفهم ليه اترفض ويتصرف |
| 10 | تحميل الصور Image download | **متاح في صفحة الفاتورة** (المكان الثالث من قرار الكارت #11) | التاجر يحمّل صور منتجاته اللي حجزها للتسويق |
| 11 | حالة البلوك Blocked state | **Full-page block** (زي Sub-Flow #3 و #4) — مفيش access، زر تواصل بس | البلوك عقوبة، رفع الحظر قرار إداري يدوي |
| 12 | الإلغاء/التعديل Cancel/edit | **مفيش من الموقع** — خدمة العملاء فقط (من BUSINESS_LOGIC) | الفاتورة immutable بعد الـ Confirm |
| 13 | الـ Bilingual | **كل الشاشات AR/EN، CSS Logical Properties، أرقام الفواتير LTR** | اتساق مع باقي الـ flows |

<a name="خريطة-الحالات"></a>
## 🗺️ خريطة حالات الفاتورة من جهة التاجر | Invoice States Map (Merchant View)

ده أهم مفهوم في الـ Sub-Flow ده. التاجر **مش بيشوف أسماء الحالات التقنية** — بيشوف لغته هو. الجدول ده بيربط الحالة التقنية باللي التاجر بيشوفه فعلاً.

This is the most important concept. The merchant **doesn't see technical state names** — he sees his own language. This table maps the technical state to what the merchant actually sees.

| الحالة التقنية Technical | اللي التاجر بيشوفه Merchant sees | اللون Color | الأكشن المتاح Action |
|---|---|---|---|
| `open` | 🟦 مفتوحة — تقدر تضيف | أزرق | أضف قطع / اطلب دفع |
| `locked` | 🟧 محتاجة دفع | برتقالي | اطلب دفع / استبدل الإيصال |
| `locked` + دفعة معلقة | 🕐 تحت المراجعة | برتقالي فاتح | استبدل الإيصال / انتظر |
| `grace` | ⏳ بتقرب تتقفل — ادفع بسرعة | أصفر | اطلب دفع (إلحاح) |
| `continuation` | 🔁 مفتوحة للاستكمال | أزرق فاتح | أضف قطع (لحد 20) / حوّل لشحن |
| `blocked` | 🚫 موقوفة | أحمر | تواصل مع الإدارة بس |
| `paid` (شحن) | ✅ مدفوعة — بتتجهز/اتشحنت | أخضر | تابع الشحن |
| `paid` (استكمال) | ✅ مدفوعة — رجعت مفتوحة | أخضر | أضف قطع / حوّل لشحن |

> ⚠️ **قاعدة لغوية مهمة:** الـ wireframes و الـ UI **لازم** تستخدم العمود التاني (لغة التاجر). أسماء الحالات التقنية للكود بس. ده متوافق مع قاعدة "لغة التاجر مش لغة المطوّر" في BUSINESS_LOGIC.
>
> ⚠️ **Important language rule:** Wireframes and UI **must** use the second column (merchant language). Technical state names are for code only.

<a name="url-structure"></a>
## 🌐 URL Structure

```
/my/invoices                  → لوحة الفواتير Invoices dashboard
/my/invoices?state=open       → مفلتر: المفتوحة Filtered: open
/my/invoices?state=payment    → مفلتر: محتاجة دفع (locked + grace)
/my/invoices?state=paid       → مفلتر: المدفوعة
/my/invoices/1042             → صفحة الفاتورة المفردة Single invoice #1042
/my/returns                   → سياسة المرتجع (الديفوهات)
```

> الفلتر بيتعكس في الـ URL (زي قرار الـ SEO في باقي الـ flows: B2B merchants share links). صفحة الفاتورة ليها URL ثابت يتشارك مع الموظفين.
>
> الفلاتر بتعمل group منطقي: "محتاجة دفع" = `locked` + `grace` مع بعض (التاجر بيفكّر "محتاجة فلوس" مش "Locked vs Grace").

### القرار: عرض السياسة + واتس (بدون فورم) | Decision: Display policy + WhatsApp (no form)

- المرتجع للديفو فقط
- الإبلاغ بصور خلال 24 ساعة من الاستلام

- شحن الديفو مناصفة بين التاجر والمكتب

<a name="security"></a>
## 🔒 Security: Defense in Depth

| الطبقة Layer | الحماية Protection |
|---|---|
| **Access control** | التاجر يشوف **فواتيره هو بس** — `partner_id` filter على كل query. محاولة فتح `/my/invoices/9999` لفاتورة مش بتاعته → 403/404 |
| **State-based actions** | الأكشن المتاح بيتحدد من الـ backend حسب حالة الفاتورة — مش من الـ frontend. مثلاً "اطلب دفع" على فاتورة Paid = مرفوض backend |
| **Blocked enforcement** | لو `is_blocked = True` → كل الـ routes بترجع شاشة الحظر (middleware/decorator)، مش بس إخفاء الأزرار |
| **Immutability** | مفيش endpoint لإلغاء/تبديل/تخفيض كمية — حتى لو حد عمل request يدوي |
| **Rejection reason** | بيتعرض من الـ backend، مفيش تعديل من جهة التاجر |

> **القاعدة:** كل الـ actions والـ access checks في الـ backend. الـ frontend بيخفي الأزرار للـ UX، لكن الأمان الحقيقي في الـ backend (زي باقي الـ flows).
